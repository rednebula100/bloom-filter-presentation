## Slide 1 — 컴퓨터는 왜 일부러 틀린 답을 내는가?

- 제목: 컴퓨터는 왜 일부러 틀린 답을 내는가?
- 부제: 블룸 필터의 거짓 양성 확률과 메모리 최적화
- 핵심 문장: 정확성 100%를 포기하면 더 빠르고 가벼워질 수 있다.

발표 대본:

> 안녕하세요. 저는 컴퓨터가 일부러 틀릴 가능성을 허용하면서도 더 효율적으로 작동하는 방법에 대해 탐구했습니다.

>
> 이번 탐구에서는 블룸 필터가 어떤 원리로 데이터를 기억하는지 살펴보고, 거짓 양성 확률을 결정하는 변수들을 수학적으로 분석하겠습니다.
>
> 또한 이론적으로 구한 결과가 실제 컴퓨터 시뮬레이션과 얼마나 일치하는지도 확인해 보겠습니다.

## Slide 2 — 완벽한 기억은 무겁다

내부 클릭: **1회**

### Beat 0 — 원본 전체 저장

- 원본 데이터를 모두 저장하는 방식이 중심이다.
- 데이터가 누적되며 저장 부담이 커지는 장면을 보여 준다.
- 오른쪽 블룸 필터 영역은 존재하지만 비활성 상태로 약하게 표시한다.
- 작은 카드들의 집합이 아닌 하나의 강한 좌우 비교 장면으로 구성한다.

대본 A:

> 먼저 블룸 필터가 왜 필요한지 생각해 보겠습니다.
>
> 어떤 데이터가 이미 저장되어 있는지 확인하려면 가장 확실한 방법은 원본 데이터를 모두 보관하는 것입니다.
>
> 하지만 데이터가 수백만 개, 수천만 개로 증가하면 저장 공간이 커지고, 데이터를 탐색하는 데에도 더 많은 자원이 필요합니다.
>
> 예를 들어 악성 인터넷 주소 검사, 이미 방문한 페이지 확인, 캐시 데이터 조회처럼 완전한 원본보다 존재 여부를 빠르게 판단하는 것이 중요한 상황이 있습니다.

**[클릭 1]**

### Beat 1 — 압축된 비트 기억

자동 연출:

1. 오른쪽 블룸 필터 영역을 활성화한다.
2. 작은 비트 배열을 왼쪽부터 순차적으로 점등한다.
3. 왼쪽의 무거운 원본 저장은 흐리고, 오른쪽의 압축된 비트 기억을 강조한다.
4. 약 520ms에 핵심 대비 문장을 표시하고, 전체 연출을 약 940ms 안에 마친다.

이 자동 연출 안에 추가 수동 Step은 없다.

대본 B:

> 블룸 필터는 이런 상황에서 원본 데이터 전체를 저장하지 않고, 0과 1로 이루어진 작은 비트 배열만 사용합니다.
>
> 따라서 메모리 사용량을 크게 줄이면서도 특정 데이터가 존재하는지를 매우 빠르게 검사할 수 있습니다.
>
> 다만 정보를 압축해서 저장하기 때문에, 그 대가로 아주 작은 오류 가능성이 생깁니다.

## Slide 7 — 거짓 양성 확률을 결정하는 수학

내부 클릭: **1회**

### Beat 0 — 거짓 양성 확률 공식

- 공식 `P ≈ (1 - e^(-kn/m))^k`를 화면 중심에 크게 표시한다.
- `m`은 비트 배열 길이, `n`은 저장한 데이터 개수, `k`는 해시 함수 개수, `P`는 거짓 양성 확률이다.

대본 A:

> 거짓 양성은 우연처럼 보이지만, 그 확률은 수학적으로 예측할 수 있습니다.
>
> 비트 배열 길이 m, 저장한 데이터 개수 n, 해시 함수 개수 k를 알면 이 근사식으로 거짓 양성 확률 P를 계산할 수 있습니다.
>
> 한 비트가 한 번의 해시에서 선택되지 않을 확률은 1-1/m이고, 총 kn번의 위치 선택 후에도 0일 확률은 (1-1/m)^(kn)입니다. 이를 지수함수로 근사하면 e^(-kn/m)이 됩니다. 따라서 비트가 1일 확률은 1-e^(-kn/m)이고, 조회 시 k개 위치가 모두 1일 확률이 거짓 양성 확률 공식이 됩니다.
>
> 이 공식은 해시 결과가 비트 배열 전체에 균등하게 분포하고, 각 위치 선택이 서로 독립적이라고 가정한 근사식입니다.

**[클릭 1]**

### Beat 1 — 변수 사이의 균형

자동 연출:

1. `n 증가 → P 증가` 관계를 표시한다.
2. 190ms 뒤 `m 증가 → P 감소` 관계를 표시한다.
3. 다시 190ms 뒤 `k 변화 → P 감소 후 증가` 관계를 표시한다.
4. 980ms에 핵심 문장을 표시한다.

이 자동 연출 안에 추가 수동 Step은 없다.

대본 B:

> 저장하는 데이터가 많아지면 비트가 더 많이 채워져 오류 확률이 커집니다. 반대로 메모리를 늘리면 충돌이 줄어듭니다.
>
> 해시 함수 개수는 단순히 많을수록 좋은 것이 아니라, 처음에는 확률을 낮추다가 어느 지점 이후 다시 높입니다.

핵심 화면 문장:

> 거짓 양성 확률은 데이터 수, 메모리 크기, 해시 함수 개수의 균형으로 결정된다.

## Slide 8 — 해시 함수는 많을수록 좋지 않다

내부 클릭: **1회**

### Beat 0 — 실험 조건과 빈 그래프

- 조건은 `m = 1000`, `n = 200`, `k = 1 ~ 10`이다.
- x축은 해시 함수 개수 `k`, y축은 거짓 양성 확률 `P`이다.
- 아직 이론 곡선과 최적값은 표시하지 않는다.

대본 A:

> 비트 배열 길이를 1000, 저장할 데이터 수를 200으로 고정하고 해시 함수 개수만 1개부터 10개까지 바꿔 보겠습니다.

**[클릭 1]**

### Beat 1 — 이론 곡선과 최적점

자동 연출:

1. 청록색 이론 곡선을 왼쪽부터 그린다.
2. `k = 3~4` 구간과 최저점을 라임색으로 강조한다.
3. 최적값 공식 `k_opt = (m/n) ln 2`와 수치 대입 결과 `≈ 3.47`을 표시한다.
4. 980ms에 핵심 문장을 표시한다.

이 자동 연출 안에 추가 수동 Step은 없다.

대본 B:

> 곡선을 보면 k가 1에서 3으로 늘어날 때 오류율이 빠르게 감소하지만, 4를 지나면 다시 증가합니다.
>
> 최적값 공식에 수치를 대입하면 약 3.47이므로, 정수로는 3개 또는 4개가 가장 효율적입니다.

핵심 화면 문장:

> 더 많이 검사한다고 항상 더 정확해지는 것은 아니다.

## Slide 9 — 이론은 실제 실험에서도 맞을까?

내부 클릭: **1회**

### Beat 0 — 이론 곡선과 실험 조건

- `m = 1000`, `n = 200`, `k = 1 ~ 10` 조건을 표시한다.
- 각 조건마다 미삽입 데이터 10,000개를 조회하고, 서로 다른 고정 seed 5개의 결과를 평균한다.
- 오른쪽 그래프에는 이론 곡선만 먼저 표시한다.

대본 A:

> 이제 이론이 실제 컴퓨터 실험에서도 비슷하게 나타나는지 확인해 보겠습니다.
>
> 각 k마다 넣지 않은 데이터 10,000개를 조회하고, 서로 다른 고정 seed 5개로 반복한 뒤 거짓 양성률의 평균을 사용했습니다.

**[클릭 1]**

### Beat 1 — 시뮬레이션 평균 비교

자동 연출:

1. 실제 블룸 필터 시뮬레이션의 5회 평균 점을 k=1부터 순서대로 이론 곡선 위에 추가한다.
2. `k = 3~4` 부근의 최저점을 강조한다.
3. 980ms에 핵심 문장을 표시한다.

이 자동 연출 안에 추가 수동 Step은 없다. 실시간 랜덤 실행 버튼은 만들지 않는다. 화면에 표시하는 관측값은 실제 비트 배열과 double hashing을 사용하는 결정론적 시뮬레이션이 계산하며, 삽입 데이터와 조회 데이터는 서로 다른 namespace로 만들어 겹치지 않는다.

대본 B:

> 실험값은 표본의 차이 때문에 이론값과 완전히 같지는 않지만, 전체적인 곡선의 모양은 매우 비슷합니다.
>
> 특히 반복 실험에서도 k가 3개에서 4개일 때 가장 낮은 오류율이 나타납니다.

핵심 화면 문장:

> 반복 실험에서도 이론과 비슷한 최저점이 나타난다.

시뮬레이션 평균은 코드에 관측값 배열로 저장하지 않고, 고정 seed `104729, 130363, 155921, 196613, 262147`의 실제 실행 결과에서 계산한다.

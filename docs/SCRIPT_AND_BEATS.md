# 발표 대본과 Manual Beats

## Slide 1 — 컴퓨터는 왜 일부러 틀린 답을 내는가?

내부 클릭: **0회**

### Beat 0 — 표지

초기 화면부터 다음 내용을 모두 표시한다.

- 제목: 컴퓨터는 왜 일부러 틀린 답을 내는가?
- 부제: 블룸 필터의 거짓 양성 확률과 메모리 최적화
- 핵심 문장: 정확성 100%를 포기하면 더 빠르고 가벼워질 수 있다.

발표 대본:

> 안녕하세요. 저는 컴퓨터가 일부러 틀릴 가능성을 허용하면서도 더 효율적으로 작동하는 방법에 대해 탐구했습니다.
>
> 일반적으로 컴퓨터는 항상 정확한 답을 내야 한다고 생각하기 쉽습니다. 하지만 실제 컴퓨터 시스템에서는 완벽한 정확성보다 속도와 메모리 효율이 더 중요한 경우도 있습니다.
>
> 그 대표적인 사례가 블룸 필터입니다.
>
> 이번 탐구에서는 블룸 필터가 어떤 원리로 데이터를 기억하는지 살펴보고, 거짓 양성 확률을 결정하는 변수들을 수학적으로 분석하겠습니다.
>
> 또한 이론적으로 구한 결과가 실제 컴퓨터 시뮬레이션과 얼마나 일치하는지도 확인해 보겠습니다.

첫 번째 오른쪽 입력은 애니메이션을 진행하지 않고 Slide 2 / Beat 0으로 이동한다.

## Slide 2 — 완벽한 기억은 무겁다

내부 클릭: **1회**

### Beat 0 — 원본 전체 저장

- 원본 데이터를 모두 저장하는 방식이 중심이다.
- 데이터가 누적되며 저장 부담이 커지는 장면을 보여 준다.
- 오른쪽 블룸 필터 영역은 존재하지만 비활성 상태로 약하게 표시한다.
- 작은 카드들의 집합이 아닌 하나의 강한 좌우 비교 장면으로 구성한다.

대본 A:

> 먼저 블룸 필터가 왜 필요한지 생각해 보겠습니다.
>
> 어떤 데이터가 이미 저장되어 있는지 확인하려면 가장 확실한 방법은 원본 데이터를 모두 보관하는 것입니다.
>
> 하지만 데이터가 수백만 개, 수천만 개로 증가하면 저장 공간이 커지고, 데이터를 탐색하는 데에도 더 많은 자원이 필요합니다.
>
> 예를 들어 악성 인터넷 주소 검사, 이미 방문한 페이지 확인, 캐시 데이터 조회처럼 완전한 원본보다 존재 여부를 빠르게 판단하는 것이 중요한 상황이 있습니다.

**[클릭 1]**

### Beat 1 — 압축된 비트 기억

자동 연출:

1. 오른쪽 블룸 필터 영역을 활성화한다.
2. 작은 비트 배열을 왼쪽부터 순차적으로 점등한다.
3. 왼쪽의 무거운 원본 저장은 흐리고, 오른쪽의 압축된 비트 기억을 강조한다.
4. 약 520ms에 핵심 대비 문장을 표시하고, 전체 연출을 약 940ms 안에 마친다.

이 자동 연출 안에 추가 수동 Step은 없다.

대본 B:

> 블룸 필터는 이런 상황에서 원본 데이터 전체를 저장하지 않고, 0과 1로 이루어진 작은 비트 배열만 사용합니다.
>
> 따라서 메모리 사용량을 크게 줄이면서도 특정 데이터가 존재하는지를 매우 빠르게 검사할 수 있습니다.
>
> 다만 정보를 압축해서 저장하기 때문에, 그 대가로 아주 작은 오류 가능성이 생깁니다.

다음 오른쪽 입력은 Slide 3 / Beat 0으로 이동한다.

## Slide 3 — 데이터를 저장하지 않고 위치를 저장한다

내부 클릭: **1회**

### Beat 0 — 입력과 해시 위치

- 입력값은 `apple`이다.
- H₁, H₂, H₃를 한 줄에 배치한다.
- 0~15 인덱스를 가진 16칸 비트 배열을 크게 표시한다.
- 모든 비트는 0이다.
- 입력에서 해시 함수로 향하는 구조는 보이지만, 결과 경로와 활성 비트는 아직 표시하지 않는다.
- 입력 → 해시 함수 → 비트 배열의 방향을 2초 안에 이해할 수 있게 한다.

대본 A:

> 블룸 필터에 apple이라는 데이터를 넣는 과정을 살펴보겠습니다.
>
> 블룸 필터는 입력값을 여러 개의 해시 함수에 통과시킵니다.
>
> 해시 함수는 데이터를 특정 범위의 숫자로 변환하는 함수입니다. 여기서는 apple이 각각 3번, 8번, 14번 위치로 변환되었다고 가정하겠습니다.

**[클릭 1]**

### Beat 1 — 위치만 기억

자동 연출:

1. H₁에서 3번 비트로 이어지는 경로를 표시한다.
2. 200ms 뒤 H₂에서 8번 비트로 이어지는 경로를 표시한다.
3. 다시 200ms 뒤 H₃에서 14번 비트로 이어지는 경로를 표시한다.
4. 650ms부터 3, 8, 14번 비트를 130ms 간격으로 0에서 1로 바꾼다.
5. 1000ms에 핵심 문장을 표시하고, 전체 연출을 약 1380ms 안에 마친다.

이 자동 연출 안에 추가 수동 Step은 없다.

대본 B:

> 계산된 위치의 비트는 0에서 1로 변경됩니다.
>
> 여기서 중요한 점은 블룸 필터 안에 apple이라는 문자열 자체는 저장되지 않는다는 것입니다.
>
> 블룸 필터는 데이터 자체가 아니라, 그 데이터가 가리킨 위치만 기억합니다.
>
> 이 방식 덕분에 매우 작은 공간으로 많은 데이터의 존재 여부를 표현할 수 있습니다.

핵심 화면 문장:

> 데이터 자체가 아니라  
> 데이터가 가리킨 위치만 기억한다.

## 검증할 입력 흐름

1. 시작: Slide 1 / Beat 0
2. 오른쪽: Slide 2 / Beat 0
3. 오른쪽: Slide 2 / Beat 1
4. 오른쪽: Slide 3 / Beat 0
5. 오른쪽: Slide 3 / Beat 1
6. 왼쪽: Slide 3 / Beat 0
7. 왼쪽: Slide 2 / Beat 1
8. 왼쪽: Slide 2 / Beat 0
9. 왼쪽: Slide 1 / Beat 0
